## Workflow Architecture

**Input:** Plain-language descriptions in English or Russian covering architectural, physical,
optimization, and GUI changes to a Hodgkin-Huxley neuron simulator (PySide6 + Numba).

**Output:** Code written by the agent, then reviewed via:
- Manual read + run the program
- Review skill in Windsurf
- Validation audit in Google AI Studio (Gemini)

**Agent pattern:**
- Primarily **single-agent, multi-step prompts** — one agent per task session
- Occasionally **2 agents in parallel** when changes are isolated (e.g., core vs. GUI with no overlap)
- Sometimes **ping-pong**: task handed between agents for cross-validation or continuation
- Agent selection depends on task complexity and model strengths

**Tools available:** Read, Write, Browser, CMD — everything except direct git commands.

## Quality & Evaluation

**Evaluation methods:**
- Manual testing and code reading
- Running the simulation, observing behavior
- Validation test suite (tests + rules stored in a project directory)
- Comparison against published literature data
- Benchmarks (performance targets for Numba/home PC)
- AI review (Windsurf review skill, Gemini audit)
- Agents have a skill to search and validate scientific literature

**Common failure modes (ranked by risk):**
1. **Silent physics drift** — agents historically fitted biophysical parameters to match expected output rather than literature values, undermining scientific validity. Mitigated post-skill-introduction.
2. **Desync across layers** — changes to core calculations not propagated to GUI, analytics, or docs; new features not reflected in all three layers simultaneously.
3. **Skipped validation** — agents sometimes omit running validation tests after changes.
4. **Stale documentation** — old docs not removed, new state not documented.
5. **Minor code issues** — forgotten imports, isolated one-line bugs. Low severity.

**Highest-risk areas:**
- Computational core (Numba-vectorized solver)
- Large refactors and optimizations
- Tightly coupled subsystems: physics ↔ analytics ↔ GUI ↔ docs

## Constraints

**Cost:**
- Claude Pro → minimize token usage; prefer concise prompts and targeted edits
- GPT → rate-limited (weekly + 5h), but better utilized for long uninterrupted task batches
- Kiki → unlimited; no cost concern

**Latency:** Not critical. Agents may run for up to 2 hours on complex tasks.
Avoid: infinite task loops, excessively long test suites during iteration.

**Compliance:** None. Personal open-source project. No data privacy restrictions.

| 3 | **Safety** | Protect computational core integrity; prevent silent regressions |

**Primary problem to solve:**
> Maintain consistency of architecture, integrity of physics, and highest biophysical validity —
> while keeping GUI pleasant and comprehensible, and performance extreme enough for home machines.
>
> The core risk is *silent inconsistency*: physics changing without GUI/docs/tests following,
> or parameters drifting from literature values without detection.

## Workflow Control Plane

Shared workflow routing, prompt structure, context budgets, validation gates, and safety guards are
centralized in `.ai/WORKFLOW_CONTROL_PLANE.md`.

Agents should load that file before non-trivial work, then load only the domain-specific files needed
for the task. This replaces the previous "operate within defaults" context habit with an explicit
budget and handoff contract.

**Routing convention:**
- Start with a single senior agent for most implementation tasks.
- Use parallel agents only when write scopes are isolated and the handoff contract is explicit.
- For large audits, collect parallel findings before any fixer edits shared files.

**Promotion gates:**
- Preset/channel/core-calculation changes require branch-contour tests and deterministic artifacts.
- GUI-only changes use focused compile/smoke checks unless they touch physics or presets.
- Documentation-only changes must point to current canonical files and avoid stale duplicate claims.
